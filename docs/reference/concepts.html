Architectural overview of the maf codebase. Read this before diving into any specific file.

1. The flow

A maf command runs in four phases:

  1. Macro expansion (load time) — maf-defcmd parses the declaration and generates a defun that wires the other three phases together.
  2. Resolve (call time) — given point and the calc stack state, produce a context alist describing what the command should operate on and how to commit the result.
  3. Body (call time) — the user's code. Receives clean values via bindings, computes a result, calls commit.
  4. Commit (call time) — applies the body's result to the calc buffer per the context's instructions.
maf-defcmd declaration                        (load time)
    │
    ▼
macro expansion → defun
    │
    ▼ (call time)
resolve → context alist ──┐
    │                     │
    ▼                     │
body (sees bindings)      │
    │                     │
    ▼  val                ▼
commit ─── reads context, applies val

Resolve and commit are peer concerns: resolve reads state into the context, commit writes state from the context. They share the context schema documented at the top of maf-resolve.el.

2. The target system

Resolve dispatches to one of five targets based on where point is and what calc state is active. In priority order:

Target When it fires Operates on
selection An active calc selection exists anywhere on the stack The selected sub-formula
home Point is past the last stack entry (at or below .) The top stack entry
subexpr Point is inside an entry's formula text The sub-formula at cursor
equation Point is on a stack entry that is a relation (, !, …) LHS and RHS (body runs once per side)
entry Point is at line-prefix or EOL of a stack entry The whole formula

The cascade lives in maf--resolve-context. The predicates live in maf-lib.el (maf--at-*-p) and maf-sel.el (maf--sel-any-p).

Each target has a dedicated maf--resolve-target-X function that produces the target-specific portion of the context alist.

The equation target also fires by conversion: when home, entry, or subexpr resolves an :expr that is itself a relation, the context is rewritten to equation after the fact (see Relation mapping).

3. The context alist

Everything resolve learns is carried in one flat alist — the context — which the later phases read but never mutate. maf--resolve-context assembles it from three layers, appended in order, so earlier keys shadow later ones:

  1. Target keys — from the matched maf--resolve-target-X function, possibly converted by relation mapping (below).
  2. Declaration options — the defcmd's OPTS merged in verbatim. Known keys are :arity (the symbol unary or binary), :prefix (the trail label string), :map (-1 opts out of relation mapping), and the I/H variant links (command symbols); any other keyword passes through untouched, so resolve/commit extensions can define their own.
  3. Ambient state — snapshots taken at resolve time: :keep (the calc-keep-args-flag boolean) and :point (an alist of point's placement, captured before the rewrite destroys it — see Point restoration for its keys).

For example, a binary mult command (:prefix "mult") invoked with point on the x of 2: 6 x + 12, with 5 at the top of the stack, resolves to: 

((:target    . subexpr)                ; target keys
 (:expr      . (var x var-x))
 (:expr-ref  . (var x var-x))          ; the encased cons from the entry
 (:arg       . 5)
 (:m         . 2)
 (:commit-m  . 2)
 (:commit-n  . 1)
 (:post-pop  . 1)
 (:reselect  . nil)
 (:arity     . binary)                 ; declaration options
 (:prefix    . "mult")
 (:keep      . nil)                    ; ambient state
 (:point     . ((:pos . 7) (:line . 1) (:affinity . nil))))

Two consumers read it. The macro-generated command wires the body:

  • :target — a symbol: home, selection, subexpr, equation, or entry. Selects the execution shape (single body run vs once per side); commit also dispatches on it to pick the splice path.
  • :expr — the clean operand, a normalized calc formula, bound as the body's EXPR.
  • :arg — the second operand for binary commands, a normalized calc formula, bound as ARG; nil for unary.
  • :rel-op, :lhs, :rhs — equation target only. :rel-op is the relation's head symbol (calcFunc-eq, calcFunc-neq, calcFunc-lt, calcFunc-leq, calcFunc-gt, or calcFunc-geq); :lhs and :rhs are the two sides as normalized formulas. The body runs once with EXPR bound to :lhs, once to :rhs, and the two results are reassembled under :rel-op before a single commit.

maf--commit applies the result:

  • :commit-m, :commit-n, :post-pop — integers encoding the push/pop recipe: the stack level to push at, the entries popped there first, and the entries popped from the top afterward (see The commit recipe).
  • :m — integer stack level (1 = top) of the target entry; where commit fetches the full formula for splicing.
  • :expr-ref — the encased cons cell for the eq-based splice. Selection and subexpr only (see Encasing).
  • :reselect — boolean: whether the committed result becomes the new selection.
  • :prefix — string: trail label recorded with the operation.

The commit instructions are computed at resolve time, not by commit: each target function knows how its own result should land (including the keep-args variants), and commit just executes the recipe. That split is what keeps commit target-agnostic apart from the splicing special case.

The authoritative key-by-key schema lives at the top of maf-resolve.el.

3.1. Point restoration

Calc's command epilogue parks point at home after every command. The :point snapshot is what undoes that: it is consumed after commit, as the generated command's last act — restore has to run once calc-wrapper returns, because anything done inside the wrapper is clobbered by the epilogue.

The snapshot (maf--point-snapshot) is itself an alist of three keys:

  • :pos — integer: point's buffer position.
  • :line — integer: the line number at point.
  • :affinity — a symbol classifying the position semantically: home when point was at or below the . line, eol at end of line, bol in the line-number prefix, or nil inside an entry's formula text.

maf--point-restore applies it:

  • home affinity restores nothing; calc's default placement already matches.
  • Otherwise point returns to :pos, corrected back to :line if the rewrite shifted it, and eol=/=bol affinity is re-applied on the line rather than the exact position.

Line-based restore is correct for every target because of a commit invariant: commits only pop entries at the target's level and below it in the buffer (:post-pop consumes the stack top, which is displayed below the target), so the target entry's line never moves.

One case restores by anchor rather than position: point on a glyph the resolved sub-formula renders itself — its operator, a comma, its function name. Resolve records the glyph's index as :point-anchor, commit returns where the committed node landed (its cons and stack level), and point goes to that node's same structural glyph, wherever the rewrite moved it. Commuting sin(y) = k from the = leaves point on the = of k = sin(y), and the anchor survives the glyph itself changing (commuting x < y parks point on the >). It also holds through relation mapping — factoring from the = re-anchors on the rebuilt relation's = — and spaces count as glyphs, since a juxtaposed product renders its multiplication as nothing but a space: commuting 2 (4 + 3 x) from that space lands after the ) of (4 + 3 x) 2. Atoms have no anchor, and anything the anchor can't locate afterwards \(non-flat renderings, a consumed entry) falls back to the positional restore. The node-to-text coordinate machinery lives in core/maf-comp.el. Sticking point to the same atom when the formula reshapes around it is deliberately deferred.

3.2. Relation mapping

A relation can become the subject without point sitting on a relation entry: the top entry at home, an entry target at the margin, or the relation node itself under point as a subexpr. maf--resolve-map-relation inspects every home, entry, and subexpr context after resolution; when its :expr is a relation, it prepends the equation keys — :target, :rel-op, :lhs, :rhs — onto the context. The base target's commit fields remain (only :target is shadowed), so the rebuilt relation replaces exactly what the base target would have replaced.

Commands whose body consumes or produces relations whole (solve, mapeq, the relation builders) opt out with :map -1, which also disables the direct equation branch of the cascade.

4. Encasing

Calc's mechanism for distinguishing duplicate atoms by eq.

Consider 1 + 1 = (+ 1 1) in calc. The two 1=s are the same integer object — both =eq and equal. If the user selects "the first one", you can't tell them apart structurally.

calc-encase-atoms (in calc's calc-sel.el) walks the formula and replaces each naked atom with (cplx atom 0) — a fresh cons cell each time: 

(+ 1 1)    →    (+ (cplx 1 0) (cplx 1 0))
                ;; ^^^^^^^^^^  ^^^^^^^^^^
                ;; cons #A     cons #B   ← now distinguishable by eq

Calc displays (cplx N 0) as just N (complex with zero imag), so the user sees nothing different.

calc-prepare-selection calls this on a stack entry and mutates the entry in place via setcar. So once selection-related machinery runs on an entry, that entry's atoms are encased until something normalizes.

This matters for us in two places:

  1. The body shouldn't see encased values. Most calcFunc-* functions dispatch on type, and (cplx ...) inputs route to complex-number branches that produce floats. (E.g., calcFunc-sqrt on (cplx 16 0) gives a float instead of the integer 4.) So resolve exposes a clean :expr via math-normalize.
  2. Commit's splice is eq-based. calc-replace-sub-formula matches by eq, and only the original encased cons matches the cell in the entry's formula. So resolve also exposes :expr-ref — the actual cons cell — for commit to use.

5. :expr vs :expr-ref

The body and commit operate on the same sub-formula but at different points in the data flow:

  • :expr — clean form, for the body to compute with
  • :expr-ref — encased cons, for commit's eq-based splice

They're only different for selection and subexpr (the targets that splice into a parent formula). home and entry don't have :expr-ref — they operate on whole entries, no sub-formula targeting needed.

6. Reselect

When commit pushes the new entry, should the result be re-selected?

  • selection target: yes (:reselect t). User explicitly set a selection; preserve their intent by re-selecting the new sub-formula.
  • subexpr target: no (:reselect nil). User just had a cursor position; no explicit selection to preserve.
  • home / entry / equation: not applicable (no sub-formula selection).

Commit reads :reselect and passes val-encased (or nil) as the SELS arg to calc-pop-push-record-list.

7. Keep-args

calc-keep-args-flag is calc's mode where operations don't consume their inputs — the result is pushed above them rather than replacing them.

Resolve snapshots the flag as :keep and derives commit instructions accordingly:

Target Normal mode Keep-args mode
home Pop arity entries, push result at top Push result at top, no pop
selection Replace selected entry in place Push result on top, leave entry
subexpr Replace entry in place (sub-formula spliced) Push spliced result on top
entry Replace entry in place Push result on top

In schema terms, :commit-m, :commit-n, and :post-pop encode the recipe; commit just executes it.

8. The commit recipe

Three context keys describe what commit should do:

  • :commit-m — stack level to push at (the M arg to calc-pop-push-record-list).
  • :commit-n — entries to pop at :commit-m before pushing (the N arg). This is the "replace in place" knob — 1 means "pop the old, push the new", 0 means "insert on top of it."
  • :post-pop — entries to pop from the top after the push. This consumes the binary arg in non-keep mode.

Visualization for binary selection at :m = 2 (non-keep): 

Before:                              commit-m=2, commit-n=1, post-pop=1
1: arg                ──┐
2: selected entry  ──┐  │
                     │  │
                     ▼  ▼
After push at m=2 (pop 1 at level 2, push new):
1: arg
2: new (with spliced sub-formula)
                     │
                     ▼
After post-pop 1 from top:
1: new

Two operations, one push and one optional post-pop. See maf--commit-push.

9. Binary on selection-like targets

Selection, subexpr, entry: each can support binary commands, but the "second operand" needs to come from somewhere. Convention: the top stack entry serves as :arg.

This creates a self-reference problem when the target itself is the top:

  • selection / subexpr — reject. If the selected sub-formula is in the top entry, the arg would be the entry containing the sub-formula. No coherent commit semantic. Resolve errors out.
  • entry — ergonomic shift. If point is at the line-margin of the top entry and arity is binary, :m is shifted from 1 to 2. The top becomes the arg, the entry below becomes the target. User doesn't have to position point on the operand whose value will change.

Choice reflects the difference between "I pointed at this exact sub-formula" (selection/subexpr — error keeps the user honest) and "I'm near this entry" (entry — shift is friendlier).

10. Inverse / Hyperbolic variants

Calc's I and H prefix flags reroute a defcmd to a declared variant command (the :inverse, :hyperbolic, and :inverse-hyperbolic options — e.g. I on mafcmd-ln runs mafcmd-exp). Dispatch happens before any context is resolved, and the flags are consumed first, so a variant that itself declares variants does not dispatch again — that is what keeps cross-linked pairs like ln ↔ exp from looping. A set flag with no matching variant signals user-error and still consumes the flags, so the next command starts clean.

11. Contextual digit entry

The digit keys (and _, ., #, @) start calc's normal number entry, but where the result lands depends on point at the first keypress (maf-digit-start in src/minibuffer.el). On a numeric leaf the entered number replaces it; on any other sub-formula it multiplies it, number on the left, with the product left literal so nothing distributes; on a relation node it multiplies both sides. At home, in the line prefix, or at EOL the number is pushed onto the stack exactly as in plain calc — as it is in algebraic mode, for entries that escape to algebraic, and for interval entry (..). The entry minibuffer is calc's own (calc-digit-map), so the in-entry keys — e, _, :, n — are unchanged.

File Role
maf.el Package entry point, maf-mode, load order

debug/maf-dev-init.el Init file for the dedicated maf-dev instance
tests/ Step tests; a file passes when no step errors

Load order in maf.el: comp → lib → sel → hl → step → hl-verify → resolve → commit → defcmd → cmds → math → stack → bindings. Each layer depends only on earlier ones.
