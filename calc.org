* Guides and Conventions
  A command should take an explicit number of arguments - never implicit.

  For example, (my/calc-factor-by) by default works on the top 2 stack items.
  But for convenience, we may want it to to automatically take the GCD of the
  top stack item if no second argument is given.

  However, this would be a mistake. In order for (my/calc-factor-by) to work in
  context properly, it *cannot* accept an implicit number of arguments.

  Take an example where the point is over a subexpression at stack level m=3.
  The user invokes (my/calc-factor-by) to factor the subexpression. Should the
  subexpression factor by its GCD or by the stack (m=1) item? Both cases are
  valid and useful, but the choice cannot be disambiguated.

  The correct behavior is for the command to either *always* take 2 arguments
  (the target expression and the factor) or *always* take a single argument (the
  target expression - the GCD is automatically selected as the factor).

  If the first behavior is chosen, the `H` flag can be used to *explicitly*
  enable the second behavior. This is the preferred way.

* Refactor
  Context

  The macro is ~192 lines and hard to read because all dispatch logic lives inside deeply nested quasiquotes. The
  compile-time cond on opt-line/opt-map generates different ASTs for different option combinations, whole-line-form
  is a large quasiquoted fragment inlined into two branches, and wrapped-body vs body-for-map are near-identical
  variants. Goal: move all runtime dispatch into a plain Elisp function, shrinking the macro to option-parsing + thin
   glue code.

  ---
  Root causes of complexity

  1. Compile-time branching — opt-line and opt-map are compile-time constants, forcing all context dispatch into
  quasiquote soup rather than a plain cond.
  2. whole-line-form — a 6-line quasiquoted fragment computed at expansion time and spliced into two branches.
  3. wrapped-body vs body-for-map — nearly identical; differ only in whether calc-wrapper and pop-forms are appended.
  4. Equation-map body duplication — body must expand twice with different cl-flet bindings; irreducible, but can be
  isolated cleanly.

  ---
  Approach: extract my/calc-dwim--resolve

  Move all context detection and replacer construction into a new runtime function. The macro becomes a thin shell:
  1. Parses options at compile time (unchanged)
  2. Evaluates runtime inputs into let bindings
  3. Calls my/calc-dwim--resolve to get a context plist
  4. Branches on :mode — emitting body once or twice — using cl-flet to expose replace-expr

  No call-site changes (bindings, options, body syntax all preserved).

** Notes
   `start-point` and `end-point` can take the following forms:
   - h (default value, the home area, same place point is moved after calc-align-stack-window)
   - h* (anywhere in the "home" area)
   # These values are contextualized to the affected line
   - ^ (start of entry)
   - ^ + 1 (start of entry plus 1 character )
   - $ (end of entry, past the last character)
   - $ - 2(end of entry minus 2 characters)
   - p (prefix area)
   - p + 1 (prefix area plus 1 character)
   - 4 (the fourth character, alias for ^ + 3)
   - 4 (the fourth character, alias for ^ + 3)
   # These values specify the line
   - 1:4 (the fourth character of the first line)

=.ctf= files (not =.el= — calc display like =2:  3= isn't valid elisp, so a

State machine that uses =read= for s-expressions and switches to raw-text

- Stack entries pushed bottom-first via =calc-push=
- Output compared with =math-equal= / =math-normalize=
- =ctf-declare-start-cond= / =ctf-declare-end-cond= forms are evaled and
  asserted

alongside existing ERT tests with no change to the test runner invocation.

#+title: Calc Web Interface

* Calc Web Interface

A browser-based rendering layer for Emacs Calc. The browser is a dumb
terminal: it captures raw key input, sends it to Emacs over WebSocket,
and renders the calc stack as MathJax expressions. Emacs-Calc is the
sole computation engine.

** Architecture

#+begin_example
Browser (HTML/CSS/JS)
  │  keydown → ws.send({key: "Q"})
  │
WebSocket (localhost)
  │
Emacs
  ├─ key router → lookup-key + call-interactively
  ├─ calc-mode runs normally
  ├─ post-command-hook → serialize stack
  └─ ws.send({stack: [...latex strings...]})
  │
Browser
  └─ MathJax.typesetPromise() → render
#+end_example

The Emacs calc buffer remains functional in parallel. All state lives
in Emacs; the browser has none.

** Components

*** 1. Emacs WebSocket Server (~calc-web.el~)

- Minor mode ~calc-web-mode~ starts/stops the server cleanly
- ~websocket-server~ on a configurable port (default 7070)
- ~on-open~: register client, push current stack immediately
- ~on-message~: receive key event JSON, route to calc command
- ~on-close~: unregister client
- ~post-command-hook~ in calc buffer: push updated stack to all clients

#+begin_src elisp
(define-minor-mode calc-web-mode
  "Serve calc state over WebSocket."
  :lighter " CWeb"
  (if calc-web-mode
      (calc-web--start-server)
    (calc-web--stop-server)))
#+end_src

*** 2. Key Router

Receives ~{key: "Q"}~ or ~{key: "f l"}~ from browser.

- Single keys: ~(lookup-key calc-mode-map (kbd KEY))~ → ~call-interactively~
- Chord accumulation: buffer keys until a binding is found or sequence
  is invalid (clear buffer and report error)
- Modifier flags (~H~, ~I~, ~J~/option): set ~calc-hyperbolic-flag~ etc.
  before dispatching, clear after
- Unknown key: send ~{error: "unbound: KEY"}~ back to browser

*** 3. Stack Serializer

#+begin_src elisp
(defun calc-web--serialize-stack ()
  (let ((calc-language 'latex))
    (cl-loop for entry in calc-stack
             unless (eq (car entry) 'top-of-stack)
             collect (math-format-value (car entry)))))
#+end_src

Push payload: ~{stack: ["x^{2}+1", "\\frac{1}{2}", ...]}~
Stack is ordered top-first to match calc display convention.

Also serialize:
- ~calc-hyperbolic-flag~, ~calc-inverse-flag~, ~calc-option-flag~ (show
  active modifiers in the browser UI)
- Trail entry for the last operation
- Error message if last command signaled

*** 4. Browser Client (~calc-web.html~ / ~calc-web.js~)

**** Connection
#+begin_src js
const ws = new WebSocket("ws://localhost:7070");
ws.onmessage = ({data}) => render(JSON.parse(data));
#+end_src

**** Key capture
Intercept ~keydown~ on ~document~. Map browser key names to calc key
strings (~"Enter"~ → ~"RET"~, ~"Backspace"~ → ~"DEL"~, etc.). Send:
#+begin_src js
ws.send(JSON.stringify({key: calcKey}));
#+end_src

Suppress default browser behavior for keys calc uses (~"/"~, ~"*"~, etc.).

**** Rendering
Stack entries arrive as LaTeX strings. Render with MathJax:
#+begin_src js
function render({stack, flags, error}) {
  stackEl.innerHTML = stack
    .map((tex, i) => `<div class="entry">
      <span class="label">${stack.length - i}:</span>
      <span class="expr">\\(${tex}\\)</span>
    </div>`)
    .join("");
  MathJax.typesetPromise([stackEl]);
  renderFlags(flags);
  if (error) showError(error);
}
#+end_src

**** UI elements
- Stack display (main area)
- Modifier indicator bar (~HYP~, ~INV~, ~OPT~ active states)
- Error/message line
- Optional: trail panel
